# Internationalization (i18n) Architecture

This document describes how i18n is implemented in NuxtShop, including:

- Message file organization and namespaces
- The runtime store and `useI18n` composable

- User preferences and the Profile integration

## 1. Message files & namespaces

Message files live under `locales/`:

- `locales/zh-CN.ts` ŌĆō Simplified Chinese
- `locales/en-US.ts` ŌĆō English

Strings are organized by namespaces, for example:

- `ui.*` ŌĆō shared UI components (buttons, modals, toasts, empty states, etc.)
- `pages.*` ŌĆō pageŌĆælevel copy (home, cart, orders, profile, etc.)
- `profile.*` ŌĆō profileŌĆærelated sections
- `demo.*` ŌĆō texts for `/demos/*` pages
- `error.codes.*` ŌĆō messages for backend error codes

Example:

```ts
pages: {
  cart: {
    emptyTitle: 'Your cart is empty',
    emptyDesc: 'Browse the catalog and add something you like.'
  }
}
```

Keeping namespaces consistent makes it easier to locate copy and avoid key
collisions.

## 2. Runtime store & `useI18n` composable

### 2.1 Store: `stores/i18n.ts`

Responsibilities:

- Manage the current locale (e.g. `zh-CN` / `en-US`)
- Persist the locale in `localStorage` (`nuxtshop-locale`)
- Provide a single source of truth for client and server

### 2.2 Composable: `composables/useI18n.ts`

Exports:

- `t(key: string, params?: Record<string, any>)`
  - Resolves the key in the current locale
  - Supports placeholders (e.g. `{count}`) that are replaced by `params`
  - Returns the raw key if nothing is found (useful for debugging)

- `setLocale(locale: UserLanguage)`
  - Updates the storeŌĆÖs `locale`
  - Writes to `localStorage`
  - Optionally triggers a user preference update when logged in

## 3. SSR & clientŌĆæside restoration

### 3.1 Default language on SSR

- SSR uses a default locale (currently `zh-CN`) for the initial HTML;
- This avoids hydration mismatches where the server and client render
  different languages.

### 3.2 ClientŌĆæside restoration priority

On the client, the locale is chosen in the following order:

1. `localStorage['nuxtshop-locale']` if present;
2. Otherwise, infer from the browserŌĆÖs preferred language (e.g. `en` ŌåÆ `en-US`);
3. Otherwise, fall back to the default locale.

The restored locale is written back to the store and drives all subsequent
rendering.

## 4. User preference & Profile integration

### 4.1 Language selection in Profile

In Profile ŌåÆ **Preferences**, users can choose:

- Simplified Chinese (`zh-CN`)
- English (`en-US`)

When a user picks a language:

1. `setLocale` is called, updating the i18n store;
2. `nuxtshop-locale` is updated in `localStorage`;
3. If the user is logged in, `/api/user/update` is called to store
   `language` on the user document.

### 4.2 Where language preference is used

The i18n layer covers:

- Navigation (header, profile menu, cart/wishlist links)
- Auth pages (login/register)
- All major business pages:
  - Home, products, cart, orders, wishlist, profile
- Shared components:
  - `BaseModal`, `BaseConfirm`, `BaseLoading`, `BasePagination`
  - Slider and puzzle captchas
- Demo pages under `/demos/*`

This ensures the entire UI feels consistent in both languages.

## 5. Adding new localized strings

### 5.1 Steps

1. **Pick a namespace**  
   Choose a logical namespace such as `pages.checkout.*` or
   `profile.notifications.*`.

2. **Add keys to all locales**  
   - Add the key and string in `locales/zh-CN.ts`
   - Add the matching key in `locales/en-US.ts`

3. **Use `t` in components**  
   In a Vue component:

   ```vue
   <script setup lang="ts">
   const { t } = useI18n()
   </script>

   <template>
     <h1>{{ t('pages.checkout.title') }}</h1>
   </template>
   ```

4. **Use placeholders when needed**  
   Example message:

   ```ts
   // locales/en-US.ts
   pages: {
     cart: {
       itemsCount: 'You have {count} items in your cart'
     }
   }
   ```

   Usage:

   ```ts
   t('pages.cart.itemsCount', { count: cartCount })
   ```

### 5.2 Error codes

Error codes use `error.codes.<CODE>` keys, such as:

- `error.codes.AUTH_INVALID_CREDENTIALS`
- `error.codes.USER_NOT_FOUND`

`useApiErrorHandler` tries to translate these codes first and only falls back
to `message` when no translation is defined.

## 6. Summary & extension tips

NuxtShopŌĆÖs i18n setup is intentionally lightweight:

- Easy to read and extend;
- Well integrated with user preferences and error handling;
- Friendly to SSR + client hydration.

To add more languages (e.g. Japanese or Traditional Chinese):

- Create a new file under `locales/` (e.g. `ja-JP.ts`);
- Extend the `UserLanguage` type and the i18n store to include the new locale;
- Add a new option in the Profile preferences;
- Optionally, adjust SSR defaults based on user or request headers.
